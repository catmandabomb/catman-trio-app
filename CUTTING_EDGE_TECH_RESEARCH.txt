I. PDF RENDERING & VIEWING
================================================================================

1. WebGPU-Accelerated PDF Rendering
────────────────────────────────────
Status: Experimental / Frontier territory

WebGPU is the successor to WebGL, providing modern GPU access (Vulkan, Metal, D3D12).

Browser Support:
  - Chrome 113+ (stable since April 2023)
  - Edge 113+ (same Chromium base)
  - Firefox: behind flag (dom.webgpu.enabled), Nightly only
  - Safari: partial in Technology Preview; NOT stable
  - NO iOS support (WebKit-only on iOS)

PDF Rendering Applicability:
  - No production WebGPU PDF renderer exists yet — genuinely bleeding-edge
  - Key opportunity: PDF vector paths (Bezier curves, fills, strokes) via compute shaders
  - Vello (formerly piet-gpu, Linebender project) is the most relevant: GPU-first 2D
    vector renderer in Rust using compute shaders, targets WebGPU via wgpu
  - Theoretical pipeline: Parse PDF -> extract vector ops -> Vello/WebGPU compute -> rasterize

Verdict: NOT YET. No iOS support kills it for PWA. Monitor for 2026-2027.


2. PDF.js Alternatives & Next-Gen Viewers
──────────────────────────────────────────

PDF.js (Mozilla) — Still the King:
  - Current: 4.x series
  - Recent: ES modules internally, OffscreenCanvas support, improved fonts/annotations
  - ~500KB core, ~2.5MB full viewer
  - Best option for vanilla JS PWAs

PDFium WASM (Google's C++ engine compiled to WebAssembly):
  - Same engine as Chrome's native PDF viewer
  - Projects: nicbarker/pdfium-wasm on GitHub
  - Renders to pixel buffer in WASM, blits to canvas
  - Advantages: Handles edge cases PDF.js struggles with
  - Disadvantages: Large WASM binary (~10MB, ~3MB Brotli), no built-in UI/text layer
  - License: Apache 2.0
  - Works everywhere including mobile Safari

MuPDF WASM (Artifex):
  - mupdf.js — official WASM build
  - Very high fidelity, ~8MB binary
  - License: AGPL (toxic) or commercial — AVOID for this project

Comparison Table:
  | Engine       | Size     | Fidelity  | Text Select | License    | WASM? |
  |-------------|----------|-----------|-------------|------------|-------|
  | PDF.js      | ~500KB   | Good      | Yes (native)| Apache 2.0 | No    |
  | PDFium WASM | ~10MB    | Excellent | Manual layer| Apache 2.0 | Yes   |
  | MuPDF WASM  | ~8MB     | Excellent | Manual layer| AGPL/Comm  | Yes   |
  | PSPDFKit    | ~15MB    | Excellent | Yes         | Commercial | Yes   |


3. OffscreenCanvas for PDF Rendering
─────────────────────────────────────
Status: Baseline Widely Available (March 2023). ALL browsers including iOS Safari 16.4+.

THE SINGLE MOST IMPACTFUL UPGRADE AVAILABLE TODAY.

How it works:
  1. Main thread creates <canvas>, calls canvas.transferControlToOffscreen()
  2. Transfers OffscreenCanvas to Web Worker via postMessage
  3. Worker runs PDF.js rendering (CPU-heavy) directly to that canvas
  4. Result appears on screen without blocking main thread AT ALL

Pattern:
  // Main thread
  const canvas = document.getElementById('pdf-canvas');
  const offscreen = canvas.transferControlToOffscreen();
  const worker = new Worker('pdf-render-worker.js');
  worker.postMessage({ canvas: offscreen, pdfData }, [offscreen]);

  // Worker
  importScripts('pdf.min.js');
  onmessage = async (e) => {
    const pdf = await pdfjsLib.getDocument({ data: e.data.pdfData }).promise;
    const page = await pdf.getPage(1);
    const ctx = e.data.canvas.getContext('2d');
    await page.render({ canvasContext: ctx, viewport }).promise;
  };

Caveat: Once transferred, main thread cannot draw to that canvas.


4. Incremental / Streaming PDF Rendering
─────────────────────────────────────────

Linearized PDFs:
  - Web-optimized PDF with first page at file start
  - Page 1 renders immediately while rest streams in
  - Create with: qpdf --linearize
  - BIGGEST "FREE" PERFORMANCE WIN — pre-process your PDFs

PDF.js Streaming:
  - Already supports HTTP Range requests and progressive loading
  - Can begin rendering page 1 before full download
  - Google Drive supports range requests

ReadableStream + PDF:
  - Streams API pipe fetch response into incremental PDF parser
  - PDF.js has basic support for this pattern


5. Experimental Browser APIs for Document Rendering
────────────────────────────────────────────────────

View Transitions API (Chrome 111+, Safari 18+):
  - Smooth animated transitions between PDF pages
  - document.startViewTransition() for native-feeling page flips

content-visibility: auto (Baseline 2024):
  - Pages outside viewport skip layout/paint entirely
  - Massive performance for multi-page scroll mode

CSS contain: strict:
  - Prevents layout thrashing when one page re-renders

scheduler.postTask() (Chrome 94+):
  - Priority-based task scheduling
  - Visible page at user-blocking priority, pre-render at background

createImageBitmap():
  - Off-main-thread image decode/resize
  - For WASM: render to ImageData in worker, create ImageBitmap, transfer

Canvas2D Hints:
  - { desynchronized: true } — skip compositor sync, reduce latency by 16ms
  - { willReadFrequently: true } — keep in CPU memory, avoid GPU readback


6. GPU-Accelerated Text Rendering
──────────────────────────────────

Current: Canvas fillText() uses CPU text shaping — bottleneck on text-heavy pages

Approaches:
  - SDF (Signed Distance Field) fonts: pre-compute distance fields, GPU fragment shader
    Used by Mapbox GL JS. Production-proven.
  - MSDF (Multi-channel SDF): sharper corners than SDF
  - Glyph atlas caching: rasterize each glyph once, cache in GPU texture, composite
  - Vello: GPU compute shaders for glyph outlines (requires WebGPU)

Practical: SDF is proven but massive engineering effort for PDF. Better to use
OffscreenCanvas to move existing rendering off main thread.


7. Vector vs Rasterization Advances
────────────────────────────────────

Current (PDF.js): Canvas2D rasterization. Re-renders on zoom.

Vello/GPU-Compute Future:
  - GPU processes all paths in parallel
  - 10,000-path page: 50ms CPU -> 2ms GPU
  - "Render at any zoom in real-time" becomes feasible
  - Status: works in native Rust, WebGPU port not production-ready

Pathfinder (Mozilla Research):
  - GPU vector rendering via WebGL (not just WebGPU)
  - More mature than Vello for web, but development slowed

SVG Rendering: PDF.js REMOVED this backend. Too slow for complex pages. Avoid.


PDF TIER SUMMARY
────────────────

  1. OffscreenCanvas + Web Worker rendering (biggest win, universal support)
  2. content-visibility: auto (free perf for multi-page)
  3. Linearized PDFs (pre-process for instant first-page)
  4. createImageBitmap() for efficient WASM output transfer

Tier 2 — Near-Term (6-12 months):
  5. PDFium WASM (replace PDF.js engine, keep text layer overlay)
  6. scheduler.postTask() (priority rendering)

Tier 3 — Future (12-24 months):
  8. WebGPU compute-shader rendering (when Safari ships stable WebGPU)

Avoid:
  - MuPDF WASM (AGPL license)
  - SVG rendering path (too slow)
  - WebGPU today (no iOS)

1. AudioWorklet — The Foundation
────────────────────────────────
Status: Baseline Widely Available (April 2021). All browsers including iOS Safari 14.5+.

What's cutting-edge now:
  - WASM inside AudioWorklet: C/C++/Rust DSP at near-native speed on audio thread
  - SharedArrayBuffer ring buffer: lock-free data exchange between threads
    (requires Cross-Origin-Isolation headers — breaks Google Drive/Fonts)
  - AudioWorkletProcessor.port for non-realtime comms

For Catman Trio: AudioWorklet is THE correct metronome implementation.
Sample-accurate, immune to main-thread jank.


2. WebCodecs API — AudioDecoder/AudioEncoder
─────────────────────────────────────────────
Status: Limited. Chrome 94+, Edge 94+, Safari 16.4+ (partial). NO FIREFOX.

Enables streaming decode without loading entire file into memory.
Supported codecs: MP3, AAC, Opus, FLAC, Vorbis, PCM (varies by browser).

Verdict: Not viable as primary path (Firefox gap). Good progressive enhancement.

6. Metronome Precision — Full Spectrum
──────────────────────────────────────

Level 1 — Classic lookahead (current implementation):
  setTimeout on main thread, schedule OscillatorNode.start(time) ahead.
  Works but vulnerable to main-thread jank.

Level 2 — AudioWorklet tick generator (RECOMMENDED):
  process() called every 128 samples (~2.9ms at 44.1kHz).
  Maintain sample counter, generate click at beat boundary.
  IMMUNE to main thread jank. Post message for UI flash.

Level 3 — AudioWorklet + pre-rendered click buffer:
  Transfer custom click AudioBuffer to worklet, copy samples on beat.
  Custom sounds with sample-perfect timing.

Level 4 — WASM DSP in AudioWorklet:
  Rust/C++ metronome compiled to WASM. Overkill for metronome alone.

Level 5 — scheduler.postTask() (Chrome):
  Priority scheduling for Level 1 pattern. Not a replacement for AudioWorklet.

BEST PRACTICE 2025: Level 2 or 3 in AudioWorklet.
Pattern:
  1. AudioWorkletProcessor counts samples, generates/copies click at beat boundaries
  2. Posts { type: 'beat', beat: n } to main thread via port.postMessage
  3. Main thread flashes UI indicator
  4. Audio perfectly timed; UI flash ~5-10ms variance (perceptually fine)


7. Audio Streaming / MediaSource Extensions
────────────────────────────────────────────
Limited availability. Safari uses ManagedMediaSource (17+).
Recent: MediaSource in Workers, FLAC in MSE, low-latency MSE.

Verdict: Overkill for song-length files (3-7 min). Only relevant for very long
recordings or live streaming.


AUDIO TIER SUMMARY
──────────────────

  1. AudioWorklet metronome (sample-accurate, universal since 2021)
  2. Opus audio format (smaller files, better quality, now universal)
  3. setSinkId() (route metronome to headphones — Chrome/Edge progressive)

IV. STORAGE, ENCRYPTION & SYNC
================================================================================

1. Origin Private File System (OPFS)
─────────────────────────────────────
Status: Baseline March 2023 — all browsers.

Private, origin-scoped virtual file system. No permission prompts. Sync access
in Web Workers via createSyncAccessHandle().

  const root = await navigator.storage.getDirectory();
  const fh = await root.getFileHandle('data.bin', { create: true });

  // Main thread: async
  const w = await fh.createWritable(); await w.write(data); await w.close();

  // Worker: sync (MUCH faster)
  const sh = await fh.createSyncAccessHandle();
  sh.write(buffer, { at: offset }); sh.flush(); sh.close();

Use cases: SQLite-WASM on OPFS, crash-resilient write buffer, large blob cache.
No 5MB limit (subject to overall quota).


2. Storage Buckets API
──────────────────────
Status: Chrome 122+ only. No Firefox/Safari.

Named buckets with independent eviction policies:
  const bucket = await navigator.storageBuckets.open('critical-data', {
    persisted: true, durability: 'strict', quota: 50 * 1024 * 1024
  });

Verdict: Chrome-only. Design with concept in mind, use navigator.storage.persist().


3. Web Locks API
────────────────
Status: Baseline March 2022 — all browsers.

Named async locks for cross-tab coordination:

  // Leader election (only one tab syncs)
  await navigator.locks.request('sync-leader', { ifAvailable: true }, async lock => {
    if (!lock) return; // another tab is leader
    await runSyncLoop();
  });

  // Exclusive write lock
  await navigator.locks.request('write-queue', async lock => {
    await flushWriteQueue();
  });

VERY HIGH value. Eliminates race conditions between tabs for write queue.
Use for: leader election, exclusive write locks, shared read locks.


4. Compression Streams API
───────────────────────────
Status: Baseline May 2023 — all browsers.

Native gzip compression/decompression. Zero dependencies:

  // Compress before encryption
  const compressed = new Blob([data]).stream()
    .pipeThrough(new CompressionStream('gzip'));

  // Chain: data -> compress -> encrypt -> push to GitHub
  async function compressThenEncrypt(jsonString, aesKey, iv) {
    const compressed = await new Response(
      new Blob([jsonString]).stream().pipeThrough(new CompressionStream('gzip'))
    ).arrayBuffer();
    return crypto.subtle.encrypt({ name: 'AES-GCM', iv }, aesKey, compressed);
  }

QUICK WIN: JSON compresses 60-80%. Reduces GitHub data branch size dramatically.


5. WebCrypto Advances — X25519 / Ed25519
──────────────────────────────────────────
Status: Chrome 113, Firefox 130, Safari 17.

Modern elliptic curve crypto. Ed25519 for signatures, X25519 for key agreement.

X25519 enables end-to-end encrypted key exchange between devices WITHOUT a server:

  // User A generates keypair
  const { publicKey, privateKey } = await crypto.subtle.generateKey(
    { name: 'X25519' }, true, ['deriveKey']
  );
  // User B derives shared AES key using A's public key
  const sharedKey = await crypto.subtle.deriveKey(
    { name: 'X25519', public: userAPublicKey },
    userBPrivateKey,
    { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']
  );

PERFECT for future per-user auth: serverless encrypted key exchange.


6. CRDTs for Conflict-Free Offline Sync
────────────────────────────────────────

Yjs (recommended):
  - Most popular CRDT framework for JS
  - ~15KB gzipped, UMD/ESM builds (no build tools needed)
  - Shared types: Y.Map, Y.Array, Y.Text
  - y-indexeddb for local persistence
  - Used by Notion, Jupyter, BlockSuite

Automerge:
  - Rust core -> WASM (~120KB)
  - JSON-like CRDT, built-in sync protocol
  - Requires WASM loading

Pattern for GitHub-backed CRDT sync:
  Device writes -> CRDT state (IndexedDB/OPFS)
                -> encode binary diff
                -> encrypt with AES-GCM
                -> push to GitHub data branch
  Other device  -> pull encrypted diffs
                -> decrypt -> merge (automatic conflict resolution)

ELIMINATES manual field-level merge conflict resolution entirely.
Two users editing same practice list merge cleanly without custom logic.

9. Content-Addressable Storage (CAS) Patterns
──────────────────────────────────────────────
Architectural pattern (no API needed):

  Store data keyed by hash of content (like Git internally).
  Benefits: deduplication, immutable = cacheable forever, integrity verification,
  Merkle trees for efficient sync.

  async function store(data, key) {
    const encrypted = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, key, data);
    const hash = bufToHex(await crypto.subtle.digest('SHA-256', encrypted));
    await opfs.write(hash, encrypted);
    return hash; // content address
  }

Combined with CRDTs: only changed data pushed, both sides verify integrity.


10. Emerging Sync Architecture Patterns
─────────────────────────────────────────

Pattern A — Git-as-Database with CRDTs:
  OPFS (local) -> Yjs CRDT -> Compress -> Encrypt -> GitHub data branch

Pattern B — X25519 Key Exchange for Multi-User:
  Users publish X25519 public keys to GitHub (unencrypted)
  Each derives shared AES key via X25519 + own private key
  No server ever sees plaintext or private keys

Pattern C — OPFS + SQLite-WASM:
  wa-sqlite on OPFS sync access handle
  Full SQL queries offline, CRDT sync layer for deltas
